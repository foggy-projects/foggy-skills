# Plan Evaluator

对技术优化计划进行可维护性评估，提供替代方案分析和风险识别。

## 使用场景

当用户需要以下操作时使用：
- 评估技术方案的可行性和可维护性
- 审查代码优化计划或架构设计
- 识别技术债务和潜在风险
- 对比不同实现方案的优缺点

## 执行流程

1. **了解项目背景**
   - 询问用户该计划要实现的需求或解决的关键问题
   - 确认项目的背景、目标、所处阶段
   - 确认项目的预期用户规模与场景
   - 如果用户已提供完整背景，跳过询问直接使用

2. **思考替代方案**
   - 技术选型：分析是否有其他更合适的技术选型
   - 架构设计：评估是否可以使用更灵活、可扩展的架构设计
   - 实现方式：寻找更简洁的实现方式，降低复杂度

3. **评估当前计划**
   - 对比当前方案与替代方案的优缺点
   - 分析实现难度、技术适配性、维护难度、可扩展性
   - 在代码简洁性和架构扩展性之间找到平衡点
   - 识别是否会引入不必要的技术栈和项目复杂度
   - 预判是否会在后续引发技术债务问题

4. **输出评估结果**（见输出格式）

## 输入要求

用户需提供：
- 优化计划或技术方案描述（必需）
- 项目背景信息（可选，如未提供则在执行流程中询问）

## 输出格式

```
✅ 计划合理：[原因] / ❌ 计划不合理：[原因]

- 替代方案：[如果不合理，给出最好的替代方案；如果合理但有更优方案，也在此说明]
- 复杂度：[分析该计划是否会引入巨大的复杂性]
- 风险点：[指出潜在的可维护性风险]
```

## 约束条件

- 必须优先选择清晰、易理解的实现方式，避免过度的框架或复杂依赖
- 避免推荐会增加不必要技术栈的方案
- 必须预见性地处理潜在的维护难题
- 评估时考虑项目所处阶段和规模，避免过度设计

## 决策规则

- 如果缺少关键背景信息 → 询问用户，确保评估准确性
- 如果当前计划存在明显过度设计 → 标记为不合理，提供简化方案
- 如果发现潜在技术债务 → 在风险点中明确指出
- 如果存在多个可行方案 → 对比分析，说明适用场景
- 如果计划会引入新的复杂依赖 → 评估是否值得，权衡收益与成本
