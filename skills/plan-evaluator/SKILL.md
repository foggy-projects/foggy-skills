# Plan Evaluator

对技术方案、执行规划或验收文档进行合理性评估，提供替代方案分析、风险识别和一致性复核。

## 使用场景

当用户需要以下操作时使用：
- 评估技术方案的可行性和可维护性
- 审查代码优化计划或架构设计
- 复核验收文档是否与原始 requirement、plan、evidence 一致
- 识别技术债务和潜在风险
- 对比不同实现方案的优缺点

不适用：
- 编写正式验收记录
- 代替签收角色给出最终 signoff
- 只做进度落盘或版本文档分发

## 评审对象识别

在正式评审前，先识别当前输入更接近哪一类对象，并按对应策略评审。

默认分为 6 类：

1. **需求文档**
   - 例如：需求说明、场景需求、产品规则、验收标准草案
2. **总规划文档**
   - 例如：根目录总规划、版本开发方案、阶段拆解方案
3. **子目录执行文档**
   - 例如：下沉到 owning repo 的 requirement、implementation plan、execution prompt、progress 模板
4. **业务代码**
   - 例如：Controller、Service、Domain、Facade、Repository、前端业务实现
5. **测试代码**
   - 例如：单元测试、集成测试、回归测试、e2e / playwright 测试
6. **验收文档**
   - 例如：acceptance、signoff、验收结论记录

如果用户没有显式说明类型，应根据输入内容自动判断：

- 以目标、背景、规则、验收口径为主，几乎没有实现细节：优先判为“需求文档”
- 以阶段、ownership、模块职责、完成门为主：优先判为“总规划文档”
- 以某个子模块开工说明、完成定义、测试要求、回写要求为主：优先判为“子目录执行文档”
- 以具体源码、类设计、实现逻辑为主：优先判为“业务代码”
- 以测试用例、断言、测试夹具、覆盖范围为主：优先判为“测试代码”

如果类型不明确，应先在输出中显式写明当前采用的评审对象类型，再给结论。

## 规则装载

识别出评审对象类型后，按需加载对应规范，不要每次把所有规则都塞进上下文。

优先读取：

- 需求文档：`references/requirement-review.md`
- 总规划文档：`references/root-plan-review.md`
- 子目录执行文档：`references/execution-doc-review.md`
- 业务代码：`references/business-code-review.md`
- 测试代码：`references/test-code-review.md`
- 验收文档：`references/acceptance-doc-review.md`

如当前任务同时跨两类对象，按主对象优先，只补读确实需要的第二份规范。

## 执行流程

1. **了解项目背景**
   - 询问用户该计划要实现的需求或解决的关键问题
   - 确认项目的背景、目标、所处阶段
   - 如果当前对象隶属于某个版本或迭代，优先确认该版本是否已有“版本目标 / 迭代目标 / 成功标准”文档；如有，先以其为评审基线
   - 如果当前是版本级规划、需求补充或验收复核，但没有明确版本目标，应优先指出“缺少迭代目标基线”，不要直接把功能清单当成版本目标
   - 如果用户已提供完整背景，跳过询问直接使用

2. **识别评审对象类型**
   - 先判断当前输入属于哪一类对象
   - 在输出里显式说明本次采用的评审对象类型

3. **按类型加载规范**
   - 只读取当前对象对应的 `references/*.md`
   - 不要无差别加载全部规范

4. **思考替代方案**
   - 技术选型：分析是否有其他更合适的技术选型
   - 架构设计：评估是否可以使用更灵活、可扩展的架构设计
   - 实现方式：寻找更简洁的实现方式，降低复杂度

5. **按对应规范完成评估**
   - 使用当前对象类型的检查矩阵做判断
   - 额外检查当前对象是否能映射到所在版本的迭代目标与成功标准
   - 识别风险、缺口、替代方案与结论一致性

6. **输出评估结果**（见输出格式）

## 输入要求

用户需提供：
- 待评审对象（需求、规划、执行文档、验收文档、业务代码或测试代码）（必需）
- 项目背景信息（可选，如未提供则在执行流程中询问）

## 输出格式

```
✅ 评估通过：[原因] / ❌ 评估不通过：[原因]

- 替代方案：[如果不合理，给出最好的替代方案；如果合理但有更优方案，也在此说明]
- 复杂度：[分析该计划是否会引入巨大的复杂性]
- 风险点：[指出潜在的可维护性风险]
- 证据缺口：[如果是验收文档，指出缺了哪些关键证据或依据]
- 结论一致性：[如果是验收文档，说明 signoff 结论是否站得住]
```

## 约束条件

- 必须优先选择清晰、易理解的实现方式，避免过度的框架或复杂依赖
- 避免推荐会增加不必要技术栈的方案
- 必须预见性地处理潜在的维护难题
- 评估时考虑项目所处阶段和规模，避免过度设计
- 如果评审对象是验收文档，只输出评审意见，不替代 `foggy-acceptance-signoff` 编写正式签收记录
- 不要把主 `SKILL.md` 继续膨胀成全量规则仓库；规则优先沉到 `references/`

## 决策规则

- 如果缺少关键背景信息 → 询问用户，确保评估准确性
- 如果没有先识别评审对象类型 → 先补类型判断，再进入结论
- 如果没有装载对应对象类型的规范 → 优先补读对应 `references/*.md`
- 如果当前计划存在明显过度设计 → 标记为不合理，提供简化方案
- 如果当前对象属于某个版本或迭代，但未说明它支撑哪个版本目标 → 标记为“与迭代目标关系不清”
- 如果当前版本已存在迭代目标文档，而需求、规划或验收结论无法回指该目标 → 标记为“与版本目标脱节”
- 如果当前计划把实现细节写得过深，已经替子 agent 做了大量局部编码设计，而不是定义流程和完成标准 → 标记为偏离规划目的
- 如果发现潜在技术债务 → 在风险点中明确指出
- 如果存在多个可行方案 → 对比分析，说明适用场景
- 如果计划会引入新的复杂依赖 → 评估是否值得，权衡收益与成本
- 如果验收结论与证据不匹配 → 明确指出不能直接签收
- 如果验收文档缺失关键依据或证据 → 优先标记为待补证，而不是勉强通过
- 如果执行规划要求编写测试但未要求运行通过 → 标记为完成定义不足，建议补充"测试必须运行通过才算完成"
- 如果方案将 Controller / Service / 编排逻辑放在部署壳（`*-cloud-service`）中 → 标记为不合理，Controller 应放对应 impl 模块或编排模块
- 如果方案中编排 Service 注入多个 impl 模块且存在循环依赖 → 标记为风险，建议创建独立编排模块
- 如果方案涉及代码归属但未交叉检查项目层级约束文档 → 标记为证据不足
- 如果文档没有在最前面说明其作用和目标读者 → 标记为文档识别性不足

## 输出补充要求

输出评审结果时，建议在开头先补一行：

`- 评审对象类型：需求文档 | 总规划文档 | 子目录执行文档 | 业务代码 | 测试代码 | 验收文档`

这样后续用户和其他 skill 可以直接看出本次结论基于哪套检查策略。
