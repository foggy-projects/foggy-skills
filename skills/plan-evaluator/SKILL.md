# Plan Evaluator

对技术方案、执行规划或验收文档进行合理性评估，提供替代方案分析、风险识别和一致性复核。

## 使用场景

当用户需要以下操作时使用：
- 评估技术方案的可行性和可维护性
- 审查代码优化计划或架构设计
- 复核验收文档是否与原始 requirement、plan、evidence 一致
- 识别技术债务和潜在风险
- 对比不同实现方案的优缺点

不适用：
- 编写正式验收记录
- 代替签收角色给出最终 signoff
- 只做进度落盘或版本文档分发

## 执行流程

1. **了解项目背景**
   - 询问用户该计划要实现的需求或解决的关键问题
   - 确认项目的背景、目标、所处阶段
   - 确认项目的预期用户规模与场景
   - 如果用户已提供完整背景，跳过询问直接使用

2. **思考替代方案**
   - 技术选型：分析是否有其他更合适的技术选型
   - 架构设计：评估是否可以使用更灵活、可扩展的架构设计
   - 实现方式：寻找更简洁的实现方式，降低复杂度

3. **评估当前对象**
   - 如果输入是技术方案或执行规划：
     - 对比当前方案与替代方案的优缺点
     - 分析实现难度、技术适配性、维护难度、可扩展性
     - 在代码简洁性和架构扩展性之间找到平衡点
     - 识别是否会引入不必要的技术栈和项目复杂度
     - 预判是否会在后续引发技术债务问题
     - **模块归属与依赖检查**（如方案涉及代码放置或模块划分）：
       - 检查每个新增文件的目标模块是否符合项目分层约束（common / impl / service 等）
       - 检查 `*-cloud-service` 等部署壳中是否被放入了 Controller、Service、编排逻辑、Form、Result 等业务代码（部署壳只应有启动类和集成测试）
       - 检查编排 Service 是否需要注入多个 impl 模块的 Bean，如果是，检查模块间依赖方向是否会产生循环
       - 如果存在循环依赖风险，检查方案是否规划了独立的编排模块（如 `*-platform-impl`）来承接跨模块编排
       - 如果项目有 CLAUDE.md 或等价的层级约束文档，以该文档为判断依据
   - 如果输入是验收文档：
     - 检查验收依据是否覆盖原始 requirement / implementation plan / acceptance criteria
     - 检查 evidence 是否足以支撑最终结论
     - 检查 blocker 是否被错误降级
     - 检查 `accepted-with-risks` 是否被滥用
     - 检查结论、风险项、后续动作之间是否自洽

4. **输出评估结果**（见输出格式）

## 输入要求

用户需提供：
- 优化计划、技术方案或验收文档（必需）
- 项目背景信息（可选，如未提供则在执行流程中询问）

## 输出格式

```
✅ 评估通过：[原因] / ❌ 评估不通过：[原因]

- 替代方案：[如果不合理，给出最好的替代方案；如果合理但有更优方案，也在此说明]
- 复杂度：[分析该计划是否会引入巨大的复杂性]
- 风险点：[指出潜在的可维护性风险]
- 证据缺口：[如果是验收文档，指出缺了哪些关键证据或依据]
- 结论一致性：[如果是验收文档，说明 signoff 结论是否站得住]
```

## 约束条件

- 必须优先选择清晰、易理解的实现方式，避免过度的框架或复杂依赖
- 避免推荐会增加不必要技术栈的方案
- 必须预见性地处理潜在的维护难题
- 评估时考虑项目所处阶段和规模，避免过度设计
- 如果评审对象是验收文档，只输出评审意见，不替代 `foggy-acceptance-signoff` 编写正式签收记录

## 决策规则

- 如果缺少关键背景信息 → 询问用户，确保评估准确性
- 如果当前计划存在明显过度设计 → 标记为不合理，提供简化方案
- 如果发现潜在技术债务 → 在风险点中明确指出
- 如果存在多个可行方案 → 对比分析，说明适用场景
- 如果计划会引入新的复杂依赖 → 评估是否值得，权衡收益与成本
- 如果验收结论与证据不匹配 → 明确指出不能直接签收
- 如果验收文档缺失关键依据或证据 → 优先标记为待补证，而不是勉强通过
- 如果方案将 Controller / Service / 编排逻辑放在部署壳（`*-cloud-service`）中 → 标记为不合理，Controller 应放对应 impl 模块或编排模块
- 如果方案中编排 Service 注入多个 impl 模块且存在循环依赖 → 标记为风险，建议创建独立编排模块
- 如果方案涉及代码归属但未交叉检查项目层级约束文档 → 标记为证据不足
