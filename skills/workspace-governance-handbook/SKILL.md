# Workspace Governance Handbook

当用户希望为一个包含多个工作目录的项目建立或更新 `CLAUDE.md` 体系时使用本 skill。

目标不是假定项目一定是微服务，而是建立一种稳定的协作方式：
- 根目录负责总控、规划、验收、E2E、BUG 回流、全局调整
- 子目录负责具体开发、局部验证、结果回写
- 输出文件有稳定落点
- 已有 `CLAUDE.md` 以安全合并为主，不直接覆盖

## 1. 使用边界

适用：
- 单仓多模块
- 多个相关 Git 项目共处一个工作区
- 前后端分层项目
- 只有后端或只有前端的项目
- 有独立测试目录的项目
- 需要为根目录和子目录建立协作手册

不适用：
- 只有单一目录、没有子工作目录的极小项目
- 用户只想改某一个现有 `CLAUDE.md` 的一句话，不涉及体系设计

## 2. 先做识别，不要先套微服务模板

先扫描当前工作区，识别目录角色，再决定是否需要独立 handbooks。

优先识别这些角色：
- `root-controller`：总控层，负责规划、验收、版本文档、跨目录协调
- `backend-module`：后端业务目录、服务目录、SDK 目录
- `frontend-module`：前端应用、管理台、H5、组件库
- `web-aggregator`：Web 聚合层，可选
- `test-suite`：E2E、集成测试、回归测试目录
- `shared-library`：公共库、协议、SDK、工具包
- `related-repo`：同一工作区下的独立 Git 项目，但与主项目有业务关系

尤其注意：
- 不要因为有多个目录，就默认它们是微服务
- `related-repo` 和 `shared-library` 在很多项目里都应成为独立 agent 工作目录
- 没有 `web` 层是正常情况

需要进一步判断目录角色时，按需阅读：
- 项目结构样例：[references/topologies.md](references/topologies.md)

## 3. 三种操作模式

根据用户目标选择一种模式：

### `audit`
- 只分析当前工作区结构
- 给出建议的目录角色、是否需要独立 agent、是否需要 handbooks
- 不写文件

### `scaffold`
- 为缺失的目录新建 `CLAUDE.md`
- 根目录和子目录都可生成

### `merge`
- 更新已有 `CLAUDE.md`
- 默认增量合并，不直接重写整个文件

如果用户没有明确说模式：
- 先默认 `audit`
- 用户确认后再进入 `scaffold` 或 `merge`

## 4. 根目录与子目录的职责分离

### 根目录 `CLAUDE.md` 应保留的内容
- 项目定位
- 当前工作区职责
- 版本规划规则
- 子任务下沉规则
- 验收与 BUG 回流流程
- 输出文档命名规则
- 子 agent 提示词要求
- 独立 agent 工作目录清单
- 多 Git 项目之间的协作原则

### 子目录 `CLAUDE.md` 应保留的内容
- 目录定位
- 开发目标
- 继承的上层约束
- 当前目录结构
- 工作方式
- 输出文件规则
- 与上层任务的配合规则
- 简化版边界说明
- 验证要求
- 最终回复规范
- 当前目录对“共享类型 vs 场景专属类型”的判定规则
- 完成后需要执行的 review / audit / acceptance skill
- 测试必须执行并通过的要求
- progress 回写与最终报告要求

如果项目希望做轻量版本可见性，可在根目录和子目录规则中加入：
- 当前迭代新增的 Entity、类、方法，默认补 `@since {当前版本}` 注释
- 存量代码默认不强制追补
- 只记录首次引入版本，不维护“最后一次修改版本”

模板骨架按需阅读：
- [references/templates.md](references/templates.md)

## 5. 已有 CLAUDE.md 的合并原则

对已有项目，默认使用 `merge` 思路：
- 不直接覆盖现有 `CLAUDE.md`
- 优先保留项目特有约束
- 只补齐缺失的标准章节
- 对冲突项，先指出冲突，再建议替换
- 如果文档已经很长，优先删重复内容，而不是继续叠加

推荐维护的受管区块：
- `## 输出文件规则`
- `## 与上层任务的配合规则`
- `## 最终回复规范`

不要随意重写这些内容：
- 项目业务背景
- 架构硬约束
- 团队已有命名习惯
- 版本目录约定

详细策略按需阅读：
- [references/merge-policy.md](references/merge-policy.md)

## 6. 输出文件规则

默认采用 `A + B`：

### A. 上层任务显式传入路径
- 如果上层任务已经明确给出输出文件路径，严格使用该路径

### B. 未传入时走目录默认落点
- 后端/共享库目录默认写 `开发进度` 或 `修复反馈`
- 前端目录可额外写 `体验反馈`
- 测试目录默认写 `测试反馈` 或 `回归反馈`

这些路径必须参数化，不要写死成某个项目名。至少向用户确认：
- 版本目录根路径
- 版本号格式
- 优先级前缀规则
- 是否存在专门的 `bugs/` 目录

同时建议在版本文档中统一要求：

- 所有任务文档开头先写 `## 文档作用`
- execution 类文档主要定义流程、边界、完成标准，不替子 agent 设计过细代码落点
- 代码最终放置位置由执行 agent 按当前目录结构和本模块 `CLAUDE.md` 判断

## 7. 推荐工作流

1. 扫描工作区目录和 Git 边界
2. 判断哪些目录应成为独立 agent 工作目录
3. 确认根目录是否作为总控层
4. 检查哪些目录已有 `CLAUDE.md`
5. 选择 `audit` / `scaffold` / `merge`
6. 输出建议、冲突项、待确认项
7. 得到用户确认后再落文件

在设计子目录 `CLAUDE.md` 时，优先补入这些通用执行标准：

- 先读上游文档和当前模块 `CLAUDE.md`
- 优先遵守目录现状和模块约束，而不是照搬上游文档里的示例路径
- 新增测试必须实际运行并通过
- 若测试无法运行，必须在 progress 中写清原因，不得标记完成
- 完成后必须回写 progress，并输出最终报告
- 如项目采用标准评审链路，明确：
  - `foggy-implementation-quality-gate`
  - `foggy-test-coverage-audit`
  - `foggy-acceptance-signoff`

## 8. 交付要求

最终回复至少要说明：
- 识别出的目录角色
- 建议建立独立 handbooks 的目录
- 本次采用的模式
- 新建或更新了哪些文件
- 仍需用户确认的规则

如果你实际修改了文件：
- 明确说明哪些目录是新建 `CLAUDE.md`
- 哪些目录是合并更新
- 是否保留了已有项目特有规则

## 9. 默认保守策略

除非用户明确要求，否则：
- 不批量改目录结构
- 不迁移版本文档目录
- 不自动删减已有长文档
- 不一次性覆盖所有 `CLAUDE.md`
- 不在 `CLAUDE.md` 中硬编码大量具体类名或包路径

先给结构识别结果和合并建议，再逐步落地。
