# Root Plan Review

用于评审根目录总规划、版本开发方案、阶段拆解方案。

## 重点检查

- 阶段划分是否合理
- ownership 是否明确
- 前置依赖与阻塞条件是否写清
- 发布门槛、迁移门槛、上线门槛是否明确
- 模块职责是否自洽
- 是否过度替子 agent 设计具体类、包、Form/Result 落点
- 完成定义是否包含测试执行通过、progress 回写、后置评审（quality-gate / test-coverage-audit / acceptance）
- 规划是否只管流程、边界和完成门，把实现自由度留给子 agent

## 常见否决点

- 阶段顺序错误或互相打架
- 关键前置条件遗漏
- 模块边界不清
- 规划写成”编码说明书”，偏离总规划职责
- 完成定义只有”编写测试”而没有”测试必须运行通过”
- 缺少后置评审链路（quality-gate → test-coverage-audit → acceptance）

## 输出倾向

- 重点评价结构、顺序、边界、门槛
- 默认不深入评论局部类设计，除非会直接导致规划失真
- 如果规划细化到了包路径和类命名级别，明确标记”偏离规划职责，建议下放给子 agent”
